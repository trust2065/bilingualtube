太棒了，看來你準備好直接捲起袖子改 code 了。既然這個專案是基於 WXT 框架（從 `wxt.config.ts` 可以看出來）並使用 TypeScript 開發，我們可以把你的需求拆解解成以下五個開發階段：

### Phase 1: 環境建置與核心邏輯定位
1. **Clone 與安裝**：執行 `git clone`，接著用 `pnpm install` 安裝依賴套件。
2. **啟動開發模式**：執行 `pnpm dev`，這會啟動熱更新（HMR），讓你改 code 時可以直接在瀏覽器看到結果。
3. **尋找切入點**：
   * 找出處理 YouTube 字幕的核心檔案（通常在 `entrypoints/content` 或 `lib/` 目錄下）。
   * 觀察它是如何攔截 YouTube 原始字幕，以及如何把第二行字幕塞進 DOM 裡面。

### Phase 2: CSS 樣式覆蓋 (對應你的點 2)
1. **獨立樣式檔**：找出擴充功能注入字幕的 DOM 結構（通常會有自訂的 class name），在專案中寫一個專屬的 CSS 或 SCSS 檔來覆蓋它。
2. **模仿 YouTube 原生設定**：你可以先寫死一套自己順眼的設定（例如字型對齊 YouTube 的 `Roboto, Arial, sans-serif`、加上半透明黑色背景框等）。
3. **預留選項介面 (Future)**：先在 `entrypoints/options` (如果有這個資料夾) 預留一個設定區塊，未來可以接上像是顏色選擇器、字體大小滑桿等 UI 元件，並將設定值存入 `chrome.storage.local`。

### Phase 3: 語言偵測與「僅限英文」邏輯 (對應你的點 3 & 4)
雖然你提到可以用前幾行字幕來辨識語言，但 YouTube 其實有更直接的做法：
1. **讀取 YouTube Meta Data**：研究 YouTube 播放器 API 或頁面上的 `ytInitialPlayerResponse` 物件，裡面通常會直接帶有當前字幕軌的語言代碼（例如 `en`、`en-US`）。這比比對字串更準確且節省效能。
2. **實作攔截器**：
   * 寫一個判斷式：`if (sourceLanguage !== 'en') return;`
   * 當偵測到原始字幕不是英文時，直接中斷翻譯 API 的請求，並阻止渲染第二行字幕的 DOM。

### Phase 4: 繁體中文防呆與翻譯開關 (對應你的點 4)
1. **自動隱藏邏輯**：
   * 延伸 Phase 3，如果偵測到影片的預設語系或當前字幕是繁體中文（`zh-TW`），除了不呼叫翻譯 API 外，也要確保擴充功能的 UI 不會多疊一層空白或重複的字幕上去。
2. **手動翻譯開關 (Toggle)**：
   * **UI 實作**：在 YouTube 播放器控制列（或擴充功能的 Popup 選單）塞入一個開關按鈕。
   * **狀態管理**：點擊開關時，切換狀態並寫入 `chrome.storage`。當狀態為 `disabled` 時，暫停所有字串翻譯和字幕渲染邏輯，讓畫面回歸最乾淨的 YouTube 原生狀態。

### Phase 5: 測試與打包
1. **極端情境測試**：
   * 找一部只有英文字幕的影片（測試翻譯是否正常）。
   * 找一部原生就是繁體中文的影片（測試是否自動隱藏且不發送 API 請求）。
   * 找一部沒有字幕的影片（確保程式碼不會報錯）。
2. **正式打包**：確認沒問題後，執行 `pnpm build` 產出正式版本的資料夾，然後載入到你的瀏覽器中日常使用。
